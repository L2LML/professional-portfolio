# DTW IT Center PM Toolkit — Construction & Technology Project Governance

A self-directed **project management organization system** — a reusable set of PM templates, plus two fully worked example projects — prepared as candidate work for an **Infrastructure / IT Project Manager** role at a Metro Detroit Airport IT Center.

This is not a record of an actual Metro Detroit Airport / Wayne County Airport Authority engagement. It's an applied demonstration of the governance structure the author would run on comparable projects: one **construction/infrastructure** delivery and one **technology** delivery, both scoped around the same fictional facility so the two case studies read as a coherent, connected program rather than two unrelated exercises.

## The scenario

The IT Center needs a new, resilient data center/NOC facility, and the network and operations systems that will live inside it need modernizing. That splits naturally into the two projects an infrastructure PM role is expected to run back-to-back or in parallel:

| | Project 1 — Construction/Infrastructure | Project 2 — Technology |
|---|---|---|
| **Folder** | [`/01-construction-infrastructure-project`](./01-construction-infrastructure-project) | [`/02-technology-project`](./02-technology-project) |
| **What it delivers** | A ~9,000 sq. ft. data center/NOC building addition — sitework, structural shell, N+1 electrical/mechanical, fire suppression, physical security, and full commissioning | Core network refresh, redundant fiber backbone, and migration of baggage/passenger-processing/access-control systems into the new facility, plus monitoring, cybersecurity hardening, and end-user training |
| **Duration** | 14 months | 9 months (kicks off once Project 1 hands over the facility) |
| **Budget** | $4.25M GMP | $2.65M |
| **PM disciplines shown** | GC/subcontractor management, AHJ permitting & inspections, commissioning (Cx Levels 1–4), safety, capital-budget governance | Vendor/systems-integrator management, change-control cutover planning, cybersecurity/technical-standards compliance, end-user training & hypercare support |

Project 2 is written as the direct continuation of Project 1 — it migrates into the facility Project 1 builds — which is why both charters cross-reference each other's milestones and handover point.

## What's in each project folder

Every project (and the blank template set) uses the same three-document structure:

1. **`01_Charter_...docx`** — Project Charter (purpose, objectives, KPIs, scope, milestones, assumptions/constraints, budget summary, approvals) + Stakeholder Communication Management Plan + a training/commissioning/cutover plan tailored to the delivery type.
2. **`02_PM_Tracking_Workbook.xlsx`** — the working tracker, with one tab each for: WBS/Schedule, Budget Tracker (live formulas), RACI Matrix, Stakeholder Register, Risk Register (auto-scored), Issue & Change Log, and Vendor/Contractor Register.
3. **`03_Status_and_Closeout_Report.docx`** — a recurring status-report template (RAG status, budget/schedule snapshot, top risks, escalations) followed by a final closeout report (baseline-vs-actual, lessons learned, sign-off).

## [`/templates`](./templates)

The same three-document set, blank and instructional (`[bracketed]` placeholders, a yellow-highlighted example row in every workbook tab, and a legend on the Instructions tab) — reusable for any future project, construction or technology.
